# 기술적 결정

## 2026-03-06: 프로젝트 초기 설정

### Supabase 프로젝트 공유
- **결정**: PompCore와 동일한 Supabase 프로젝트 사용
- **이유**: 사용자 데이터 통합 관리, 인프라 비용 절감

### Tailwind CSS 3 선택
- **결정**: Tailwind CSS 3 + @tailwindcss/forms 사용
- **이유**: Nebula 디자인 시스템의 커스텀 토큰 관리 용이, 빠른 프로토타이핑

### Zustand 상태관리
- **결정**: Zustand 5 사용
- **이유**: 가볍고 보일러플레이트 최소화, TypeScript 친화적

### enum 대신 const object
- **결정**: TypeScript enum 대신 const object + as const 패턴 사용
- **이유**: tree-shaking 지원, 더 나은 타입 추론, 런타임 값 접근 용이

## 2026-03-06: Phase 0 기반 인프라 설계

### 예산 모델: 하이브리드 방식
- **결정**: 예산 생성 시 '가상'(자산 변동 없음) 또는 '실제'(통장 차감) 중 선택
- **이유**: 사용자마다 예산 관리 방식이 다르므로 유연성 제공

### 대시보드: 정보 중심형
- **결정**: 총자산, 수입/지출 요약, 최근 거래, 다가오는 결제를 배치
- **이유**: 핵심 재무 정보를 빠르게 파악할 수 있는 실용적 레이아웃

### 통계 차트: Nivo 라이브러리
- **결정**: Nivo 사용 (월별 막대, 자산 추이 라인, 카테고리 도넛, 통장별 분포)
- **이유**: D3 기반 React 컴포넌트, 아름다운 기본 디자인, tree-shaking 지원

### 다중 통화: 통화별 분리 표시
- **결정**: 환율 API 없이 통화별로 따로 합산하여 표시
- **이유**: 환율 변동에 의한 오차 방지, 추가 API 의존성 없음

### 정기결제 실현: 클라이언트 처리
- **결정**: 페이지 로드 시 미실현 건 확인 후 거래내역 일괄 생성
- **이유**: 추가 서버 인프라(Edge Function) 불필요, MVP에 적합

### 소프트 삭제 패턴
- **결정**: 모든 사용자 데이터 테이블에 `deleted_at TIMESTAMPTZ` 컬럼
- **이유**: 휴지통(F-010) 기능 지원, 데이터 복원 가능

### 이미지 업로드: Supabase Storage
- **결정**: Supabase Storage 사용, 저장 기간 제한 적용
- **이유**: 추가 서비스 불필요, RLS 접근 제어, 무료 티어 1GB

### 친구 기능: 1차 제외
- **결정**: 친구/소셜 기능은 1차 개발에서 제외, 플레이스홀더만 배치
- **이유**: 핵심 재무 기능에 집중, 소셜 기능은 개발 범위가 큼

## 2026-03-06: Phase 4 마무리

### SEO: 환경변수 기반 조건부 로딩
- **결정**: Clarity/AdSense 스크립트를 `%VITE_*%` 환경변수로 주입, 미설정 시 자동 비활성
- **이유**: 개발 환경에서 불필요한 외부 스크립트 로딩 방지, 배포 시에만 활성화

### JSON-LD: WebApplication 스키마
- **결정**: `WebApplication` 타입의 JSON-LD 구조화 데이터 사용
- **이유**: AI 검색엔진(GEO) 대응, 앱 성격과 가격 정보를 명시적으로 전달

### 대시보드: 중복 fetch 통합
- **결정**: 이번달 전체 거래를 한 번만 fetch하고 recentTx를 useMemo로 파생
- **이유**: 동일 데이터를 두 번 요청하던 비효율 제거
